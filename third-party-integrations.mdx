Use this page to understand which third-party systems Revive integrates with, why they exist, where the integration is implemented, and what you need to operate it.

## Architecture overview

- **Core system of record**
  - `revive-api` uses Firebase Admin (`firebase-admin`) for Firestore and Cloud Storage.
  - `revive-admin`, `revive-dashboard`, and `revive-mobile` use Firebase client SDKs for auth + Firestore + storage access.
- **External system of record (CRM)**
  - HubSpot is used for CRM objects and workflows (contacts, deals, custom objects).
- **Common data flows**
  - Webhook-driven: third parties send events into `revive-api` endpoints under `/v1/webhooks/*`.
  - API pull: `revive-api` fetches property/valuation data from external providers on demand.
  - Client analytics/monitoring: frontends initialize analytics/monitoring vendors in their app roots.

## HubSpot

**What it is**

HubSpot is used for CRM workflows and operational automation.

**Why we use it**

HubSpot owns CRM objects (contacts, deals, custom objects) and triggers workflows that keep Revive systems in sync.

**Data flow**

- Inbound: HubSpot workflows/webhooks call `revive-api`.
- Outbound: `revive-api` calls HubSpot APIs for CRM reads/writes and transactional emails.

**Owner repo(s)**

- `revive-api` (webhook endpoints + API client)
- `revive-admin` / `revive-dashboard` (HubSpot tracking script + UI deep links)

**Relevant modules/endpoints**

- `/v1/webhooks/hubspot/createCompany`
- `/v1/webhooks/hubspot/updateCompany`
- `/v1/webhooks/hubspot/updateContractor`
- `/v1/webhooks/hubspot/invite-realtor`
- `/v1/webhooks/hubspot/production`
- `/v1/webhooks/hubspot/report-reminder/{emailId}`
- `/v1/webhooks/hubspot/revive-team`
- `/v1/webhooks/hubspot/updateRealtorEdgeUrl`
- `revive-api/src/utils/hubspotV2/hubspot.js` (HubSpot SDK client)
- `revive-api/src/utils/hubspot.js` (HubSpot mapping + associations)
- `revive-api/src/utils/transactional-emails.js` (HubSpot transactional emails)

**Operational notes**

- Webhooks are routed via `revive-api/src/webhooks/webhooks.routes.js`.
- Some webhook handlers update Firestore and then fan out push notifications.

## PandaDoc

**What it is**

PandaDoc is used for document workflows that connect into project scope and linked object updates.

**Why we use it**

PandaDoc templates and envelopes power signature and underwriting document flows.

**Data flow**

- Inbound: PandaDoc webhooks call `revive-api`.
- Outbound: `revive-api` calls PandaDoc Public API.

**Owner repo(s)**

- `revive-api`

**Relevant modules/endpoints**

- `/v1/webhooks/pandadoc/linked-objects`
- `/v1/webhooks/pandadoc/project-scope`
- `/v1/project-scopes/fill-from-pandadoc/{dealId}`
- `revive-api/src/lib/pandadoc/index.js` (PandaDoc API client)
- `revive-api/src/webhooks/webhooks.service.js` (`pandadocProjectScopeWHService`, `pandaDocLinkedObjectWHService`)

**Operational notes**

- PandaDoc webhook payloads can contain PII; API client code sanitizes/masks logs.

## Automated valuation models (AVM)

**What it is**

AVM endpoints provide property valuation and analysis capabilities.

**Why we use it**

We combine multiple providers to improve coverage and produce valuation ranges.

**Data flow**

Third-party property data providers (the specific provider may vary by environment and configuration).

`revive-api` requests provider data, then stores/updates aggregated results in Firestore (`AVM` collection) for caching.

**Owner repo(s)**

- `revive-api`

**Relevant modules/endpoints**

- `/v1/avm/`
- `revive-api/src/automated-valuation-models/avm.service.ts`
- `revive-api/src/utils/estated/index.js`
- `revive-api/src/lib/houseCanary/index.js`
- `revive-api/src/lib/rentcast/index.ts`
- `revive-api/src/lib/apify/index.ts` (Zillow + Redfin via Apify actors)

**Operational notes**

- AVM results are cached for ~30 days in Firestore.
- Zillow/Redfin data is fetched via Apify actors; outages or rate limits will impact AVM completeness.

## Firebase (auth, Firestore, Cloud Storage, FCM)

**What it is**

Firebase provides:

- Firestore for application data
- Cloud Storage for file uploads
- Firebase Auth for user identity
- FCM for push notifications

**Why we use it**

Firebase is the primary operational datastore and supports real-time client experiences.

**Data flow**

- `revive-api` uses Firebase Admin SDK for privileged access.
- Frontend apps use Firebase client SDKs.
- `revive-api` sends push notifications via FCM device identifiers stored on contacts.

**Owner repo(s)**

- `revive-api` (admin SDK + push notifications)
- `revive-admin` / `revive-dashboard` (client SDK)
- `revive-mobile` (client SDK)

**Relevant modules/endpoints**

- `revive-api/src/lib/firebase/index.js` (admin init)
- `revive-api/src/utils/push-notifications.js` (FCM send)
- `revive-api/src/media/media.service.js` (file tree metadata in Firestore)
- `revive-admin/utils/firebase.ts`
- `revive-dashboard/src/utils/firebase.ts`

**Operational notes**

- Client-side push notifications are wired in app roots (Firebase messaging `onMessage`).
