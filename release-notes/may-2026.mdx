<Update label="Week of May 18, 2026" description="Weekly recap" tags={["revive-api", "revive-dashboard", "revive-admin"]}>

A skim-friendly recap of what shipped this week. See the dated entries below for full details.

### New features

- **Revive AI Report v4.** The next-generation Vision AI Report is live across the homeowner dashboard and admin tools. v4 brings AnyProp-sourced comparables, room-aware photo grouping, and a refreshed renovation estimate flow end-to-end. For users: richer, faster reports with better comparable matches and clearer room-by-room visuals. Existing report URLs continue to work — v4 is served automatically.
- **AnyProp as a property data and imagery provider.** Comparables, property records, and home photos can now be sourced from AnyProp alongside existing providers (RestB, Zillow), with results normalized into the same response shape. For users: more reliable property coverage and better fallback when one provider is missing data. See [Property data providers](/third-party-integrations/property-data).
- **Home-imagery utility endpoints.** `POST /v1/utility/get-home-images` returns home photos for an address with automatic provider fallback, or you can pin a provider with `source`. `POST /v1/utility/deduplicate-images` removes exact and visually-similar duplicates from a set of image URLs using perceptual hashing. For users: easier to confirm a property has usable imagery before kicking off a report, and cleaner photo sets when combining sources.
- **Room-type classification on comparable photos.** Comparable images now carry per-image room types (kitchen, bathroom, bedroom, living, exterior, etc.) plus an aggregated `roomTypes` list per comparable. Results are cached after the first request, so reopening a comp is instant. For users: faster, more accurate side-by-side room comparisons in the report.

### Updates

- **Meeting scheduling polish.** The Getting Started meeting scheduler has tighter dropdown behavior, cleaner advisor selection, and fewer redundant components in the booking flow. For users: a smoother, less cluttered path to booking an advisor call. Funnels using `showcalendar=false` are unaffected — see [Intake URL parameters](/guides/intake-url-parameters#meeting-and-advisor-selection).

### Bug fixes

- **Vercel build stability.** Resolved module-resolution issues that intermittently broke dashboard and admin deployments on Vercel. For users: more reliable production releases with fewer deploy-time hiccups.

### Action required

None. The new `/v1/utility/*` endpoints are additive, and the new `source` and room-type fields on existing responses are additive. Existing integrations continue to work unchanged.

</Update>

<Update label="May 21, 2026" description="Revive AI Report v4" tags={["revive-api", "revive-dashboard", "revive-admin"]}>

### New

- **AnyProp as a property data and imagery provider.** The platform now sources comparable listings, property records, and home photos from AnyProp in addition to existing providers. AnyProp comps are normalized into the same shape as other sources, so reports, comparables, and renovation estimates work the same way regardless of where the underlying data came from.
- **`POST /v1/utility/get-home-images`.** A new endpoint returns home photos for an address. With no `source` specified it falls back across providers (RestB → AnyProp → Zillow) and returns the first successful, deduplicated set. Pass `source: "restb" | "anyprop" | "zillow"` to pin a specific provider. Used during intake to confirm a property has usable imagery before kicking off a report.
- **`POST /v1/utility/deduplicate-images`.** A new endpoint accepts an array of image URLs and returns the set with exact and visually-similar duplicates removed via perceptual hashing. Useful when combining photos from multiple sources before sending them into a render or report.
- **Room-type classification on comparable photos.** Comparable images returned by report endpoints now include per-image room types (kitchen, bathroom, bedroom, living, exterior, etc.) and an aggregated `roomTypes` list per comparable. Results are cached after the first request, so reopening a comp is instant.

### Improved

- **Revive AI Report v4.** Reports now include AnyProp-sourced comparables, room-aware photo grouping, and the updated renovation estimate flow end-to-end across the homeowner dashboard and admin tools. Existing report URLs continue to work — v4 is served automatically.
- **Meeting scheduling polish.** The Getting Started meeting scheduler has tighter dropdown behavior, cleaner advisor selection, and fewer redundant components in the booking flow.

### Fixed

- **Vercel build stability.** Resolved module-resolution issues that intermittently broke dashboard and admin deployments on Vercel.

### Action required

No action required. The new `/v1/utility/*` endpoints are additive, and the new `source` and room-type fields on existing responses are additive. Existing integrations continue to work unchanged.

</Update>

<Update label="May 9, 2026" description="API hotfix" tags={["revive-api"]}>

### Fixed

- **Admin-only Renovision renders no longer carry a homeowner `contactId`.** When `POST /v1/predictions/` is called with an `adminId` but no `reportId` — the signature of a render started from the Renovision configurator in the admin app — the AI image render job document now explicitly stores `contactId: null`. This prevents downstream consumers that read the job from associating the render with a homeowner contact. Calls that include a `reportId` (alone or alongside `adminId`) are unaffected. ([Revive-Home/revive-api#2020](https://github.com/Revive-Home/revive-api/pull/2020))

### Action required

No action required. Existing `POST /v1/predictions/` callers continue to work unchanged.

</Update>

<Update label="May 8, 2026" description="Sprint 8" tags={["revive-dashboard"]}>

### Changed

- **Getting Started GTM event simplified.** The Getting Started intake on production now pushes a `hubspot-form-success` event to the Google Tag Manager `dataLayer` with plain (non-hashed) form fields — `hsFormEmail`, `hsFormFirstName`, `hsFormLastName`, `hsFormPhone`, `hsFormName: "Revive Intake Form"`, plus `address`, `city`, and `postal_code` — so HubSpot and downstream tooling can attribute Revive AI report submissions without needing to match on hashed values. The event still fires only in production and failures are logged without blocking sign-up. ([Revive-Home/revive-dashboard#2417](https://github.com/Revive-Home/revive-dashboard/pull/2417))
- **Intake confirmation step no longer fires a duplicate GTM event.** The phone-verification confirmation step in the standalone Intake flow no longer pushes its own `hubspot-form-success` event, eliminating duplicate submissions in HubSpot. The Getting Started event above remains the single source of truth for intake attribution. ([Revive-Home/revive-dashboard#2417](https://github.com/Revive-Home/revive-dashboard/pull/2417))

### Action required

If you have GTM tags or HubSpot workflows that consume the Getting Started `hubspot-form-success` event, update them to read the plain `hsFormEmail`, `hsFormFirstName`, `hsFormLastName`, and `hsFormPhone` fields instead of the previous SHA-256 hashed `user_data` payload. Tags listening for the old `gettingStartedSubmission` event name should be retargeted to `hubspot-form-success`.

</Update>

<Update label="May 7, 2026" description="API release — Sprint 8" tags={["revive-api"]}>

### New

- **Rush requests for RMAs.** Four new endpoints let project managers flag an executed RMA as time-sensitive and route it through an approval workflow. `POST /v1/rma/{rmaId}/rush` raises a request with a required `reason` (the RMA must be in `executed` status and not already pending/approved). `POST /v1/rma/{rmaId}/rush/approve` and `POST /v1/rma/{rmaId}/rush/decline` resolve the request and require the `rmaAdmin` permission; the decline endpoint accepts an optional `reason`. `POST /v1/rma/{rmaId}/rush/remove` clears the rush flag from a previously approved RMA and also requires `rmaAdmin`. Each transition adds a note plus an activity log entry (`rush_requested`, `rush_approved`, `rush_declined`, `rush_removed`) and posts a Slack notification to the RMA channel. The RMA payload now includes `is_rush`, `rush_status` (`pending` / `approved` / `declined`), `rush_reason`, `rush_requested_by`, and `rush_requested_at`. ([Revive-Home/revive-api#2005](https://github.com/Revive-Home/revive-api/pull/2005))

### Improved

- **All HubSpot-linked contractors are returned on opportunity reads.** `GET /v1/opportunities/{id}` now resolves every contractor associated with the deal in HubSpot and returns them on a new `contractors` array. The existing `contractor` field is preserved for backwards compatibility and contains the first entry from `contractors` (or `null` if none are linked). Use `contractors` when a project has multiple contractors; continue to use `contractor` for single-contractor flows. ([Revive-Home/revive-api#2008](https://github.com/Revive-Home/revive-api/pull/2008))
- **`is_home_prep` exposed on contractor records.** The HubSpot contractor properties returned by opportunity and contractor endpoints now include `is_home_prep`, identifying contractors approved for home-prep work. The field is read directly from HubSpot — no migration required. ([Revive-Home/revive-api#2008](https://github.com/Revive-Home/revive-api/pull/2008))

### Fixed

- **RMA naming sequence skips rejected drafts.** `POST /v1/rma/` now derives the auto-generated `RMA #N` name from the count of non-rejected RMAs on the deal. RMAs with status `not_approved` no longer increment the counter, so a deal with three drafts (one rejected, two active) ends up with `RMA #1` and `RMA #2` instead of skipping a number. ([Revive-Home/revive-api#2002](https://github.com/Revive-Home/revive-api/pull/2002))

### Action required

No action required. The new rush endpoints and the `contractors` array are additive. Existing integrations that read `opportunity.contractor` continue to work unchanged.

</Update>

<Update label="May 7, 2026" description="Sprint 8" tags={["revive-dashboard"]}>

### Improved

- **GTM tracking on intake sign-up.** When a new user completes phone verification during the Getting Started intake on production, the dashboard now fires a `hubspot-form-success` event to Google Tag Manager with hashed user data (email, first name, last name, phone) so Revive AI report submissions are attributed in HubSpot and downstream marketing tooling. The event is only sent in production and failures are logged without blocking the sign-up flow. ([Revive-Home/revive-dashboard#2405](https://github.com/Revive-Home/revive-dashboard/pull/2405))
- **Project end date reflects weekly-update adjustments.** Project Scope, Public Property Page, weekly update logs, and the Progress Report flow now display `verifiedEndDate` (the date confirmed during a weekly update) when present, falling back to `endDate` otherwise. This keeps the homeowner-facing completion date in sync with the latest schedule confirmed by the project manager. ([Revive-Home/revive-dashboard#2407](https://github.com/Revive-Home/revive-dashboard/pull/2407))

### Fixed

- **Comparables modal no longer auto-applies filters.** The Vision AI Report comparables modal opens with no preset filters, so users see the full comparable set and can opt into status, list date, or similarity filters themselves. ([Revive-Home/revive-dashboard#2391](https://github.com/Revive-Home/revive-dashboard/pull/2391))
- **Sell 360 "Initial payment made" matches the gross initial check.** The deduction in the **02 Resale transaction** section of the Sell 360 transaction summary now uses `shared.initialPayment` instead of `shared.firstCheckNet`, so deductions in section 02 correctly sum to the resale transaction proceeds shown in section 01. ([Revive-Home/revive-dashboard#2402](https://github.com/Revive-Home/revive-dashboard/pull/2402))
- **TBD label preserved on custom estimate items.** The Renovation Estimate **Product** modal now propagates the `isTBD` flag from custom items, so items flagged as TBD continue to show the red `TBD` badge in the proposal view, side menu, and products list after editing. ([Revive-Home/revive-dashboard#2389](https://github.com/Revive-Home/revive-dashboard/pull/2389))

### Action required

</Update>

<Update label="May 4, 2026" description="API release" tags={["revive-api"]}>

### New

- **AI-generated weekly updates.** A new set of endpoints under `/v1/weekly-updates/ai-*` lets project managers draft homeowner-facing weekly renovation updates with Claude. `POST /v1/weekly-updates/ai-generate` produces a one-shot draft from contractor notes and photos. The conversation endpoints (`POST /v1/weekly-updates/ai-conversations`, `GET /v1/weekly-updates/ai-conversations`, `GET /v1/weekly-updates/ai-conversations/{conversationId}`, `POST /v1/weekly-updates/ai-conversations/{conversationId}/messages`, `POST /v1/weekly-updates/ai-conversations/{conversationId}/apply`) persist a per-deal, per-week chat thread so users can iterate on the draft and pin the version that was sent to the homeowner. The first message in a new conversation is automatically enriched with the previous week's applied update for continuity. All endpoints require a Firebase JWT. ([Revive-Home/revive-api#2009](https://github.com/Revive-Home/revive-api/pull/2009))

### Action required

No action required. The endpoints are additive. See [AI weekly updates](/guides/ai-weekly-updates) for usage and request/response shapes.

</Update>

<Update label="May 2, 2026" description="API hotfix" tags={["revive-api"]}>

### Improved

- **Tag AI image render jobs with an `adminId`.** `POST /v1/predictions/` accepts a new optional `adminId` field (Firebase UID of the admin who initiated the render) alongside the existing optional `reportId`. The value is persisted on the AI image render job document so admin-initiated renders — for example, jobs started from the Renovision configurator in the admin app — can be filtered and listed per admin. Both `adminId` and `reportId` can be supplied together; if a job with the same `jobId` already exists, the new identifiers are merged onto the existing record. ([Revive-Home/revive-api#2006](https://github.com/Revive-Home/revive-api/pull/2006))

### Action required

No action required. `adminId` is optional and existing `POST /v1/predictions/` callers continue to work unchanged.

</Update>
