## What it is

HubSpot is used for CRM workflows and operational automation.

## Why we use it

HubSpot owns CRM objects (contacts, deals, and custom objects) and runs workflows that keep Revive systems in sync.

## Data flow

- Inbound: HubSpot workflows/webhooks call the API.
- Outbound: the API calls HubSpot APIs for CRM reads/writes and transactional email workflows.

## Owner app(s)

- `api` (in `revive-apps`)
- `admin` (in `revive-apps`)
- `dashboard` (in `revive-apps`)

Admin tooling (Firebase JWT, admin only):

- `GET /v1/webhooks/hubspot/get-broken-phone-numbers` — scan HubSpot contacts and return any whose `phone` property is not in canonical format, with a `suggestedFix` for each.
- `POST /v1/webhooks/hubspot/fix-broken-phone-numbers` — batch-update HubSpot contacts with corrected, canonical phone numbers (`+1(NPA) NXX-XXXX`).

## Live data from HubSpot (May 2026)

Several API responses now resolve data live from HubSpot at read time rather than relying on cached values:

- **RMA pipeline stage labels.** `GET /v1/rma/{rmaId}` resolves the current pipeline stage label directly from HubSpot, so the displayed stage always matches the latest HubSpot configuration. The field is returned as `pipeline_stage_label`.
- **All linked contractors on opportunities.** `GET /v1/opportunities/{id}` now resolves every contractor associated with the deal in HubSpot and returns them on a `contractors` array. The existing `contractor` field is preserved for backwards compatibility (contains the first entry or `null`).
- **`is_home_prep` on contractor records.** Contractor objects returned by opportunity and contractor endpoints include `is_home_prep` (boolean), identifying contractors approved for home-prep work. Read directly from HubSpot — no migration required.
- **Meeting scheduling via HubSpot Scheduler API.** The intake meeting scheduler and calendar endpoints now use the HubSpot Scheduler API for availability and booking, replacing the previous direct calendar integration.

## Operational notes

- Webhook handlers typically update Firestore and may fan out internal notifications.
- When changing HubSpot field mappings, treat it as a backwards-compatible contract change.
- **Canonical phone format.** Revive stores HubSpot contact phone numbers in `+1(NPA) NXX-XXXX` (or E.164 `+1NPANXXXXXX`). Use the broken-phone-numbers admin endpoints above to scan for and repair contacts that drift out of this format — for example, after a bulk import or a manual edit in HubSpot.

## Admin tools

### HubSpot phone cleanup

The admin app exposes a password-protected tool for finding and bulk-fixing HubSpot contacts whose `phone` property fails E.164 validation. Use it when phone-format issues are blocking workflows or transactional sends.

**Where to find it**

- Standalone page: `/webhooks/hubspot-broken-phones`
- Metrics dashboard tab: **Metrics → HubSpot Phone Cleanup**

**Workflow**

1. Open the panel and authenticate with the admin password.
2. The tool calls `GET /webhooks/hubspot/get-broken-phone-numbers` to load contacts with malformed phone values, along with a `suggestedFix` for each.
3. Review the table, accept or edit the suggested fixes, and select the rows to update (single-select or bulk-select).
4. Submit. The tool calls `POST /webhooks/hubspot/fix-broken-phone-numbers` with the selected `{ contactId, phone }` updates and reports `totalRequested`, `totalUpdated`, and any per-contact `failed` entries.

**Notes**

- The fetch endpoint uses an extended timeout (~120s) because it scans contact records.
- Updates are written directly to HubSpot — there is no preview-only mode. Spot-check a few rows before running large batches.
- Failures are returned per-contact rather than aborting the batch; review the `failed` list and retry as needed.
