<Update label="May 21, 2026" description="Sprint 9 release" tags={["revive-api", "revive-dashboard", "revive-admin"]}>

### New

- **RenoVision slide in the homeowner report.** A new RenoVision slide is now part of the Revive AI Report on the homeowner dashboard, surfacing the AI-rendered "after" view of the home alongside existing report sections.
- **Comparables slide in the homeowner report.** A new comparables slide gives homeowners a focused, swipeable view of comparable properties from inside the report flow. Public report URLs work without sign-in.
- **PDF download for preliminary estimates.** Preliminary estimates (RTStay pre-estimate) now offer a one-click PDF download from the dashboard, with wider pricing columns and clearer totals on the export.
- **RTStay sharing without ARV / as-is inputs.** Sharing an RTStay project from the admin tools no longer requires the ARV and as-is values to be filled in, removing a blocker when those numbers aren't known yet.
- **Auto-fill RMA material advance from the PO sheet.** When you create or edit an RMA, the material advance amount is now populated from the linked purchase-order sheet, so project managers no longer copy the number by hand.
- **Scope item labels in weekly updates.** Custom scope-item labels now appear in the weekly update scope picker and in the published weekly update view. Homeowners see the same item names project managers use internally.
- **RMA rush access widened.** All admins can now request and remove rush on an RMA from the RMA list. Approve and decline still require the `rmaAdmin` permission. See the underlying endpoints in the May 7 entry below.

### Improved

- **Live HubSpot pipeline stage on RMAs.** RMA detail responses now fetch the pipeline stage label live from HubSpot instead of relying on a cached value, so the displayed stage always matches what's in HubSpot.
- **TBD on parent items when all sub-items are TBD.** Renovation Estimate items now show the `TBD` badge automatically when every sub-item is flagged TBD, matching the existing per-item behavior.
- **Owner's rep "Select All" persists across navigation.** The owner's rep filter on the admin RMA list keeps its "Select All" state when you navigate between pages.
- **Revive owner deal check.** The "is this a Revive-owned deal?" check now applies regardless of HIC availability and includes the Sell 360 mapped deal name, reducing false negatives on Sell 360 transactions.

### Fixed

- **Editing Revive AI reports saves correctly.** Edits made to a Revive AI Report from the dashboard now save reliably; a regression that caused some edits to be silently dropped has been resolved.
- **Apply Changes button no longer appears when only the package changes.** Switching the package on a Renovision configuration without picking new materials no longer shows a stale "Apply Changes" prompt.
- **Material selection no longer resets other materials.** Changing one material in the Renovision configurator keeps your other selections intact instead of resetting them.
- **RMA check-run cutoff respects timezone.** The check-run cutoff time for RMAs is now evaluated in the correct timezone, so RMAs near the cutoff are categorized consistently.
- **Rounded cents on financial totals.** Internal rounding on transaction totals now rounds to the nearest cent, eliminating sub-cent drift in derived figures.
- **Sidebar label and marker styling on comparables.** Renamed a comparables sidebar label and removed an unintended active-pill highlight on comparable markers in the report map.
- **Concierge ID resolution.** Corrected the Revive Concierge identifiers used in opportunity routing so concierge assignments resolve to the right user.

### Action required

No action required for existing integrations. New report slides and RMA fields are additive.

</Update>

### New features

- **Revive AI Report v4.** The next-generation Vision AI Report is live across the homeowner dashboard and admin tools. v4 brings AnyProp-sourced comparables, room-aware photo grouping, and a refreshed renovation estimate flow end-to-end. For users: richer, faster reports with better comparable matches and clearer room-by-room visuals. Existing report URLs continue to work — v4 is served automatically.
- **AnyProp as a property data and imagery provider.** Comparables, property records, and home photos can now be sourced from AnyProp alongside existing providers (RestB, Zillow), with results normalized into the same response shape. For users: easier to confirm a property has usable imagery before kicking off a report, and cleaner photo sets when combining sources.
- **Room-type classification on comparable photos.** Comparable images now carry per-image room types (kitchen, bathroom, bedroom, living, exterior, etc.) plus an aggregated `roomTypes` list per comparable. For users: a more guided way to walk through the report on phone and desktop.

- **Auto-fill RMA material advance from the PO sheet.** Material advance on an RMA is now populated automatically from the linked purchase-order sheet. For users: fewer manual copies between sheets and the RMA form.
- **Scope item labels in weekly updates.** Custom scope-item labels now appear in the weekly update scope picker and the homeowner-facing weekly update view. For users: homeowners see the same item names project managers use.

### Updates

- **Meeting scheduling polish.** The Getting Started meeting scheduler has tighter dropdown behavior, cleaner advisor selection, and fewer redundant components in the booking flow. For users: a smoother, less cluttered path to booking an advisor call. Funnels using `showcalendar=false` are unaffected — see [Intake URL parameters](/guides/intake-url-parameters#meeting-and-advisor-selection).
- **RTStay sharing no longer blocks on ARV / as-is.** Admins can now share an RTStay project without filling in the ARV and as-is values. For users: project managers can share earlier in the process.
- **RMA rush is available to all admins.** Any admin can now request or remove rush on an RMA from the list view; approve and decline remain gated on the `rmaAdmin` permission.
- **Live HubSpot pipeline stage on RMAs.** RMA detail responses fetch the pipeline stage label live from HubSpot. For users: the displayed stage always matches HubSpot.

### Bug fixes

- **Vercel build stability.** Resolved module-resolution issues that intermittently broke dashboard and admin deployments on Vercel. For users: more reliable production releases with fewer deploy-time hiccups.
- **Editing Revive AI reports saves correctly.** Fixed a regression where some edits to a report were silently dropped on save.
- **Renovation configurator selections preserved.** Switching packages no longer shows a stale "Apply Changes" prompt, and changing one material no longer resets the others.
- **RMA check-run cutoff respects timezone.** RMAs near the daily cutoff are now categorized consistently across timezones.

### Changed

### Action required

Tags listening for the old `gettingStartedSubmission` event name should be retargeted to `hubspot-form-success`.

<Update label="May 7, 2026" description="API release — Sprint 8" tags={["revive-api"]}>

### New

- **Rush requests for RMAs.** Four new endpoints let project managers flag an executed RMA as time-sensitive and route it through an approval workflow. `POST /v1/rma/{rmaId}/rush` raises a request with a required `reason` (the RMA must be in `executed` status and not already pending/approved). `POST /v1/rma/{rmaId}/rush/approve` and `POST /v1/rma/{rmaId}/rush/decline` resolve the request and require the `rmaAdmin` permission; the decline endpoint accepts an optional `reason`. `POST /v1/rma/{rmaId}/rush/remove` clears the rush flag from a previously approved RMA and also requires `rmaAdmin`. Each transition adds a note plus an activity log entry (`rush_requested`, `rush_approved`, `rush_declined`, `rush_removed`) and posts a Slack notification to the RMA channel. The RMA payload now includes `is_rush`, `rush_status` (`pending` / `approved` / `declined`), `rush_reason`, `rush_requested_by`, and `rush_requested_at`. ([Revive-Home/revive-api#2005](https://github.com/Revive-Home/revive-api/pull/2005))

### Improved

- **All HubSpot-linked contractors are returned on opportunity reads.** `GET /v1/opportunities/{id}` now resolves every contractor associated with the deal in HubSpot and returns them on a new `contractors` array. The existing `contractor` field is preserved for backwards compatibility and contains the first entry from `contractors` (or `null` if none are linked). Use `contractors` when a project has multiple contractors; continue to use `contractor` for single-contractor flows. ([Revive-Home/revive-api#2008](https://github.com/Revive-Home/revive-api/pull/2008))
- **`is_home_prep` exposed on contractor records.** The HubSpot contractor properties returned by opportunity and contractor endpoints now include `is_home_prep`, identifying contractors approved for home-prep work. The field is read directly from HubSpot — no migration required. ([Revive-Home/revive-api#2008](https://github.com/Revive-Home/revive-api/pull/2008))

### Fixed

- **RMA naming sequence skips rejected drafts.** `POST /v1/rma/` now derives the auto-generated `RMA #N` name from the count of non-rejected RMAs on the deal. RMAs with status `not_approved` no longer increment the counter, so a deal with three drafts (one rejected, two active) ends up with `RMA #1` and `RMA #2` instead of skipping a number. ([Revive-Home/revive-api#2002](https://github.com/Revive-Home/revive-api/pull/2002))

### Action required

No action required. The new rush endpoints and the `contractors` array are additive. Existing integrations that read `opportunity.contractor` continue to work unchanged.

</Update>

<Update label="May 2, 2026" description="API hotfix" tags={["revive-api"]}>

### Improved

- **Tag AI image render jobs with an `adminId`.** `POST /v1/predictions/` accepts a new optional `adminId` field (Firebase UID of the admin who initiated the render) alongside the existing optional `reportId`. The value is persisted on the AI image render job document so admin-initiated renders — for example, jobs started from the Renovision configurator in the admin app — can be filtered and listed per admin. Both `adminId` and `reportId` can be supplied together; if a job with the same `jobId` already exists, the new identifiers are merged onto the existing record. ([Revive-Home/revive-api#2006](https://github.com/Revive-Home/revive-api/pull/2006))

### Action required

No action required. `adminId` is optional and existing `POST /v1/predictions/` callers continue to work unchanged.
